# 리서치 팩트 저장소 — 오늘의딜 매거진 규칙

> **한 번 조사한 사실은 다시 조사하지 않는다. 그리고 글끼리 어긋나지 않게 한다.**
>
> 닥스노트 공통규칙 v2(26편 실측판)를 이식하고, 제조사·모델 도메인에 맞게 4가지를 개선한 판.

## 왜 필요한가

글 한 편 쓰고 조사를 버리면 두 가지가 망가진다.

1. **같은 사실을 매번 다시 조사한다.** 삼성전자서비스 문서·모델명 연식 규칙·에러코드 체계는 여러 글에 걸쳐 인용되는데 매번 새로 판다.
2. **글끼리 같은 사실을 다르게 쓴다.** ← 이쪽이 더 위험하다. 독자가 두 글을 다 읽으면 어느 쪽도 못 믿는다. 우리가 파는 건 "광고 없는 중립 가이드"라는 신뢰라, 모순은 오류보다 치명적이다.

## 구조

```
content/research/
  briefs/<brand>_<slug>.md      조사 원본 전문 (영구 보존)
  facts/<brand>_<slug>.json     재사용 가능 팩트만 추출
  INDEX.md                      자동 생성. 직접 수정 금지
```

팩트 파일을 글마다 쪼개는 이유 — 리서처 여럿이 동시에 돌아도 덮어쓰기 사고가 안 나게. 합치는 건 `node scripts/research-index.mjs`가 한다.

## 프로토콜 (반드시 이 순서)

### 0단계 — 웹 검색 전에 저장소를 먼저 읽는다

```
1. content/research/README.md      이 파일
2. content/research/facts/*.json   ★ 팩트 원본을 직접 읽을 것
3. content/research/INDEX.md       사람이 훑는 용도(참고)
```

> INDEX는 갱신이 밀리면 낡는다. **기계적 대조 기준은 언제나 `facts/*.json`이다.**
> (닥스노트 실측: INDEX를 1차 소스로 삼았더니 커버율 34%로 떨어졌고, 프로토콜을 지킨 리서처일수록 낡은 뷰를 봤다.)

### 1단계 — 이미 있는 팩트는 다시 조사하지 않는다

재사용하면 `reused_facts`에 id를 적는다.

**재사용은 많을수록 좋은 게 아니다.** 모델·연식이 안 맞으면 **안 쓰는 게 정답**이다. 읽고 안 쓰기로 판단한 것도 `declined_facts`에 이유와 함께 남긴다. **이것도 성공 사례다.**

> 예: LG 세탁기 리서처가 삼성 에러코드 팩트를 읽고 "브랜드가 달라 코드 체계가 다름"으로 거부 → 올바른 처리.

### 2단계 — 새로 조사할 것만 웹에서 확인

제조사 공식 문서를 1순위로. 공식에 없으면 사용자 후기·커뮤니티도 쓰되 `evidence: "후기"`로 기록한다.

### 3단계 — 산출물 2개를 쓴다

- `briefs/<brand>_<slug>.md` — 조사 전문
- `facts/<brand>_<slug>.json` — **이번에 새로 확보한 팩트만** (기존 팩트 복사 금지)

**자기 파일에만 쓴다.** 남의 파일·`INDEX.md`는 절대 건드리지 않는다.

### 4단계 — 앞 팩트의 오류를 발견하면 신고한다

직접 고치지 말고 자기 파일의 `disputes[]`에 적는다. 인덱스가 INDEX 최상단에 띄우고 총괄이 판정한다.

```json
"disputes": [
  { "targetId": "앞-팩트-id", "problem": "무엇이 틀렸는지", "evidence": "직접 확인한 1차 출처" }
]
```

> 실제 사례: 검색 결과 다수가 "iPhone 8은 7과 같은 버튼 조합"이라고 하지만 Apple 원문은 다르다. 이런 걸 잡으면 신고한다.

## fact 스키마

```json
{
  "topic": "이 조사의 주제",
  "brand": "삼성 | LG | 애플 | 쿠쿠 | 공통 …",
  "article": "이 팩트를 처음 쓴 글 slug (없으면 '예정')",
  "captured": "YYYY-MM-DD",
  "reused_facts": ["재사용한-id"],
  "declined_facts": [{ "id": "안-쓴-id", "why": "모델·연식이 안 맞는 이유" }],
  "disputes": [{ "targetId": "...", "problem": "...", "evidence": "..." }],
  "facts": [
    {
      "id": "kebab-case-고유id",
      "claim": "사실을 한 문장으로. 수치·코드 포함.",
      "tier": "확실 | 논쟁 | 확인실패",
      "evidence": "공식 | 후기 | 실측",
      "source": "출처 기관 + 문서명",
      "source_url": "https://…",
      "checked": "YYYY-MM-DD",
      "scope": {
        "brand": "삼성",
        "category": "세탁기",
        "line": "드럼 (그랑데 포함)",
        "years": "2015~",
        "region": "KR"
      },
      "caution": "쓸 때 주의할 점 (선택)",
      "cautionSource": "⚠️ caution에 수치·기준·연식이 있으면 필수. 그 근거의 출처."
    }
  ]
}
```

`reused_facts` / `declined_facts` / `disputes`는 **필수 필드**다. 없으면 빈 배열 `[]`을 쓴다. 필드 자체를 빼면 안 된다.
(닥스노트 실측: 26편 중 18편이 `reused_facts`를 누락해 "문서가 바뀌면 인용 글을 전부 고친다"가 기계적으로 불가능했다.)

### `cautionSource` — 출처 없는 숫자의 세탁 경로 차단

> 닥스노트 v1 최대 사고: `caution`에 source가 없어 무엇이든 주장할 수 있었고, 어디에도 근거가 없는 수치가 caution에만 존재한 채 5개 브리프로 퍼졌다. 역추적하니 출처는 **발행글 본문**이었다.
> 즉 출처 없는 숫자가 → caution → 규칙 문서 → 다음 브리프로 **세탁**됐다. 그 숫자의 진위보다 **아무도 판별할 수 없다는 것 자체가 실패다.**

**규칙: `caution`에 수치·기준·연식·코드가 등장하면 `cautionSource`가 반드시 있어야 한다.** 없으면 인덱스가 경고한다. 근거가 없으면 그 숫자를 빼라.

## tier — 확신도

| tier | 뜻 | 글에 쓸 때 |
|---|---|---|
| `확실` | 공식 문서 또는 복수 출처 일치 | 단정 서술 가능 |
| `논쟁` | 출처마다 다르거나 모델별로 갈림 | **순차 대안 톤 필수**(아래) |
| `확인실패` | 조사했으나 근거 확보 실패 | **글에 쓰지 말 것. 절차 창작 금지** |

`확인실패`도 자산이다. 다음 사람이 같은 걸 또 찾아 헤매지 않게 한다.

## evidence — 근거의 성격 (우리 신설, tier와 직교)

| evidence | 뜻 | 글에서 |
|---|---|---|
| `공식` | 제조사·기관 공식 문서 | 출처 라벨 `[공식]` |
| `후기` | 사용자 블로그·커뮤니티 보고 | 출처 라벨 `[후기]`, 본문에서 "사용자들이 보고한 방법"으로 구분 |
| `실측` | 우리가 직접 확인(링크 생존·화면 등) | 필요 시 표기 |

### ⚠️ tier=`논쟁` 또는 evidence=`후기`면 순차 대안 톤이 강제된다

부정형으로 쓰지 말고, **되는 경우를 인정하고 → 판별 기준을 주고 → 다음 수단으로 넘긴다.**

- ❌ "이 방법은 특정 모델군에서만 통하는 것으로 보고된 방법이라 공식 안내는 아닙니다"
- ⭕ "이 조합은 모델군에 따라 실제로 통합니다. **통하는 모델이라면 가장 빠른 해결책이니 먼저 눌러 보세요.** 눌러도 화면 변화가 없다면 내 모델은 조합이 다른 것이니, **앞서 설명한 전원 리셋으로 넘어가면 됩니다.**"

## facts에 넣을 것 / 넣지 말 것

**넣을 것** — 여러 글에 걸쳐 인용될 사실
- 제조사 공식 문서(URL + 무엇이 적혀 있는지) — 삼성전자서비스 solution 번호, LG 스스로해결 등
- **모델명 연식 판독 규칙** (예: 삼성 AF/AR + 숫자 뒤 알파벳 = N 2018 · R 2019 · T 2020 · A 2021 · B 2022)
- **에러코드 체계·표기 변경** (예: 구형 `E1` = 신형 `E101`, E계열↔C계열)
- 브랜드별 고객센터·보증 정책
- 표준 안전 경고 문구(냉매 충전 자격, 점프 극성, 배터리 팽창, 데이터 삭제)

**넣지 말 것**
- 그 글에만 쓰이는 1회성 세부 절차
- 이미 `facts/`에 있는 팩트의 복사본

## ⚠️ 층위 혼동 — 이 저장소를 만든 실제 이유

**같은 코드·숫자가 다른 층위에 존재할 때, 뭉쳐 쓰면 두 글이 동시에 틀린 글이 된다.**

우리가 실제로 겪은 사례:

| 사례 | 층위 | 섞으면 |
|---|---|---|
| 삼성 `dC` | 국내 공식=도어 감지 / 해외 자료=탈수 불균형 | 독자가 엉뚱한 곳을 점검 |
| 삼성 `E1` vs `E101` | 구형 2자리 표기 / 신형 3자리 표기 | 자기 코드를 표에서 못 찾음 |
| iPhone 8 강제 재시작 | 7 계열(동시 누르기) / 8 이후(3단계) | 아예 재시작이 안 됨 |
| 에어컨 자가진단 | 스탠드 2018~ / 벽걸이 2014~ / 시스템 | 없는 버튼을 찾게 됨 |

**규칙:** `scope`를 문자열이 아니라 **구조화 필드**(brand·category·line·years·region)로 적는다. 층위가 갈리는 팩트는 `caution`에 **어느 층위인지**와 **섞였을 때 무슨 허위가 되는지**를 적는다.

## 링크 부패 — 우리 도메인 특유의 위험

우리는 공식 문서 링크를 **글 하단 출처 섹션에 노출**한다. 링크가 죽으면 곧바로 품질 사고다.
`source_url` + `checked`를 반드시 적고, 인덱스를 URL 검사 모드로 주기적으로 돌린다.

```bash
node scripts/research-index.mjs --check-urls
```

죽은 링크가 나오면 역색인으로 **그 팩트를 인용한 글 전부**를 찾아 함께 고친다.

## 운영

```bash
node scripts/research-index.mjs              # 색인 생성 + 무결성 검사
node scripts/research-index.mjs --check-urls # + 링크 생존 검사(느림)
```

**검사 항목:** 중복 id / `cautionSource` 누락 / 필수 필드 누락 / scope 필드 누락 / INDEX 노후 / 미해결 `disputes` / (옵션) 링크 부패

## 실측으로 확인된 한계 (기대치 조정용)

닥스노트 26편 실측 기준, **재사용은 편당 8건 안팎에서 포화한다.** 저장소가 절감해주는 건 공통 축(우리 경우 제조사 문서·연식 규칙·코드 체계·안전 문구)뿐이고, 그건 **초반 10편이면 거의 다 캐진다.** 이후 브랜드가 갈라질수록 신규 조사가 다시 늘어난다.

**복리가 아니라 일회성 상수 절감이다.** "쌓일수록 기하급수적으로 빨라진다"고 기대하면 안 된다.
진짜 값은 조사 시간 절감보다 **모순 방지와 층위 혼동 차단**에 있다.

**첫 5편은 재사용 0건이 정상이다.** 저장소가 비어 있으니까.

## 에이전트에 박을 것

파일만 두면 지켜지지 않는다. **리서처·집필 에이전트 정의(`.claude/agents/*.md`)에 0단계를 박아야 한다.**
(닥스노트 실측: 프롬프트로만 준 항목은 26편 중 18편이 누락, 에이전트 정의에 박은 항목은 대체로 지켜짐.)
